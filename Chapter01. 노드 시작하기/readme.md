##### 1.1.1 서버.
1. 노드는 **서버 애플리케이션을 실행**하는 데 제일 많이 사용됩니다.
2. 서버는 **네트워크를 통해 클라이언트에 정보나 서비스를 제공**하는 컴퓨터 또는 프로그램을 말한다.
3. 클라이언트와 서버 그림으로 이해하기.
<img src = "https://user-images.githubusercontent.com/48505947/61341774-b953c300-a882-11e9-95c8-c96f91940efa.png">

 4. 서버라고 해서 요청에 대한 응답만 하는 것은 아니다. **다른 서버에 요청을 보낼 수도 있다.** 이때 요청을 보낸 서버가 클라이언트 역할을 한다.
 
 5. 정리해서, **서버는 클라이언트의 요청에 대해 응답**을 합니다. 응답은 **YES or No** 로 할 수 있다. 
 - Yes면 클라이언트 요청에 대한 응답을 하고, No면 클라이언트 요청에 대한 거절.

 ##### 1.1.2 자바스크립트 런타임.
 1. 노드는 자바스크립트 런타임입니다. 
  - 런타임이란? 특정 언어로 만든 프로그램들을 실행할 수 있는 환경을 뜻한다.
  - 따라서 노드는 자바스크립트 프로그램을 컴퓨터에서 실행할 수 있게 합니다.
 2. 크롬 V8 자바스크립트 엔진
  - 웹 브라우저를 만드는 데 기반을 제공하는 **오픈 소스 자바스크립트 엔진**이다.
  - 구글 크롬 브라우저와 안드로이드 브라우저에 탑재되어 있다.
 3. Libuv라는 라이브러리 **(Non-Blocking I/O 모델)** 를 사용한다.
  - **Blocking I/O 모델**
    * I/O 요청이 완료될 때까지 기다려야한다. 끝나기 전에는 함수가 반환되지 않는다.
    * CPU 자원 낭비가 심하다.
  - **Non-Blocking I/O 모델**
    * I/O 작업을 진행하는 동안 유저 프로세스의 작업을 중단시키지 않는다.
    * 유저 프로세스가 I/O처리를 위해 커널에 함수를 호출하면, 커널에서 함수의 진행 상황과 상관없이 **바로 결과를 반환한다.**
  4. 우리가 **작성한 자바스크립트 코드는 노드가 알아서 V8과 Libuv에 연결해주므로 노드를 사용할 때 C와 C++을 몰라도 된다.**

  ##### 1.1.3 이벤트 기반.
 1. 이벤트 기반이란 **이벤트가 발생할 때 미리 저장해둔 작업을 수행하는 방식.**
 2. 이벤트 리스너에 **콜백 함수를 등록한다고 표현**.
   - (ex) 클릭 이벤트 리스너에 경고 창을 띄우는 콜백 함수를 등록해두면 클릭이벤트가 발생할 때마다 콜백함수가 실행되어 경고창이 뜬다.
   - **노드도 이벤트 기반 방식으로 동작하므로 이벤트가 발생하면 이벤트 리스너에 등록해둔 콜백함수를 호출한다.**
   - <br> <img width = "650" height = "200" src = "https://user-images.githubusercontent.com/48505947/61344766-6e8b7880-a88d-11e9-8871-c9b4646656ed.jpg">
   
 3. 노드는 자바스크립트 코드에서 **맨 위부터 한 줄씩 실행된다.** 
 함수 호출 부분을 발견하면 **호출한 함수를 호출 스택에 넣는다.**
  - (ex) 스택 **(Last IN First Out)**  
  ```
  function first(){
      second();
      console.log("첫 번째");
  }
  function second(){
      third();
      console.log("두 번째");
  }
  function third(){
      console.log("세 번째");
  }
  first();
  ```
  - 실행결과

    * 이벤트 루프 : 이벤트 발생 시 호출한 콜백 함수를 관리한다.
    * 테스트 큐 : 이벤트 발생 후 호출되어야 할 **콜백 함수들이 기다리고 있는 공간.**

  ##### 1.1.4 싱글 스레드.
  1. 노드는 **싱글 스레드** 이므로 주어진 작업을 혼자서 처리해야 한다.
    - **싱글 스레드, 블로킹 모델.**
        * <br> <img width = "650" height = "300" src="https://user-images.githubusercontent.com/48505947/61346662-aba73900-a894-11e9-9003-41f953061258.jpg">
        
            * 점원 한 명이 주문을 받아 주방에 넘기고, 주방에서 요리가 나오면 손님에게 서빙을 한다. 그 후 다음 손님의 주문을 받습니다.
            * **매우 비효율적인 방식이다.**
    
- **싱글 스레드, 논블로킹 모델(노드가 채택한 방식)**
    * <br> <img src = "https://user-images.githubusercontent.com/48505947/61347247-16597400-a897-11e9-94fc-8ed51e779068.jpg">
    
        * 점원이 한 손님의 주문을 받고 주방에 주문 내역을 넘긴 뒤 다음 손님의 주문을 받는다.
        * 주방에서 요리가 완료되면 완료된 순서대로 손님에게 서빙한다.
        * **주문한 순서와 서빙하는 순서가 일치하지 않을 수 있다!** <br>

- **멀티 스레드, 블로킹 모델**
    * <br> <img width = "650" height = "300" src = "https://user-images.githubusercontent.com/48505947/61347740-03e03a00-a899-11e9-8be4-90ee41532328.jpg">
        * 점원이 한 명씩 맡아 주문을 받고 서빙한다.
        * 손님의 수가 늘어날수록 점원의 수도 늘어나고 손님의 수가 줄어드면 노는 점원이 있다는 문제가 있다.
- 프로세스와 스레드의 차이
    * 프로세스 
        * **운영체제에서 할당하는 작업의 단위**이다.
        * 프로세스 간 **메모리 등의 자원을 공유하지 않는다.**
    * 스레드
        * **프로세스 내에서 실행되는 흐름의 단위**이다.
        * 하나의 프로세스는 여러 개의 스레드를 가질 수 있다.
        * 스레드들은 **부모 프로세스의 자원을 공유한다.**<br>
          즉, **같은 메모리에 접근할 수 있다.**
    * 프로세스와 스레드.<br>
    <img width = "650" height = "300" src = "https://user-images.githubusercontent.com/48505947/61348213-c41a5200-a89a-11e9-8d49-59b6147c0392.png">
    
    * 의문점
        * **"노드는 멀티 스레드로 하는게 좋은 것이 아닌가?"** 하는 의문이 발생.
        * 노드는 스레드를 늘리는 대신, 프로세스 자체를 복사해 여러 작업을 동시에 처리하는 **멀티 프로세스 방식**을 사용.
